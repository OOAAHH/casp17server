# 部署过程中的问题与解决方案（脱敏版）

这份文档总结了这套方案在实际落地时遇到的问题，以及最终采用的解决方式。

## 1. 根路径既要做首页，又要做提交入口

### 问题

同一个域名下：

- 用户直接访问 `/` 时需要看到 landing page
- 外部系统访问 `/?...` 时又要进入提交处理逻辑
- 同时 `POST /` 也要交给后端

### 解决方案

把路由分成两类：

- 干净访问 `/`：由 nginx 直接返回静态首页
- `/submit`、带 query 的 `GET /?...`、`POST /`：由 nginx 转发给 FastAPI

这就是当前 `error_page 418 = @query_mailer` 那套 nginx 转发策略的来源。

## 2. 需要区分“人类看的摘要”和“调试看的原始日志”

### 问题

一开始只有“成功”和“失败”两条 webhook，不够精细：

- 人类真正关心的是合法请求里的 `target`、`sequence_length`、`reply_email`
- 调试时又需要完整原始请求，包括 query、body、headers、原始 `sequence`

### 解决方案

把通知拆成双通道：

- `FEISHU_WEBHOOK`：只发合法请求摘要
- `FEISHU_REJECTED_WEBHOOK`：所有请求都发原始日志

同时把落盘状态拆成：

- `summary_status`
- `raw_status`

这样 replay 也可以按通道重试。

## 3. 调用方需要更明确的响应

### 问题

只返回 `OK` 或简单报错不够：

- `GET` 的调用者希望快速看到人类可读信息
- `POST` 的调用者更适合拿结构化 JSON

### 解决方案

把响应分成两种：

- `GET`
  - 成功：`200 text/plain`
  - 失败：`400 text/plain`
- `POST`
  - 成功：`200 application/json`
  - 失败：`400 application/json`

并且都明确返回：

- 是否 accepted/rejected
- 失败原因
- UTC 时间
- 联系邮箱

## 4. 调用方既有单行序列，也有 FASTA

### 问题

外部请求并不总是只传单行序列：

- 有的传单行序列
- 有的传标准 FASTA
- 有的把 FASTA header 和序列一起塞进 `sequence`

### 解决方案

在归一化阶段支持 FASTA 风格：

- 去掉第一行 `>` 头
- 去掉换行和空白
- 只保留最后的纯序列内容做校验

## 5. 一部分调用方传的不是“真换行”，而是字面量 `\n`

### 问题

一些外部系统虽然说自己在发 FASTA，但实际发送的是：

```text
>Target|\nGCCCGGA...
```

这里的 `\n` 是两个普通字符，不是真换行。

如果直接按 FASTA 处理，整段会被当成 header 去掉，最后得到空序列。

### 解决方案

在归一化最前面先做兼容转换：

- `\r\n` -> 真换行
- `\n` -> 真换行
- `\r` -> 真换行

然后再走 FASTA 归一化。

这样下面三种都能通过：

- FASTA 真换行
- FASTA 字面量 `\n`
- FASTA 字面量 `\r\n`

## 6. 历史请求里还存在错误参数格式

### 问题

历史请求里出现过少写 `=` 的片段，例如：

```text
&targetTargetName&
```

### 解决方案

在参数解析里增加对已知 alias 前缀的兼容：

- 如果识别到 `targetTargetName`
- 自动按 `target=TargetName` 解释

## 7. 前端玻璃效果在静态背景下不明显

### 问题

首页一开始只有静态背景，玻璃边缘和折射感不够明显。

### 解决方案

前端最终采用了这套思路：

- 玻璃主容器 + 内层轨道层
- 轻微鼠标/滚动视差
- hover 下划线细动画
- 按钮 hover 改为灰白强调，而不是强烈换色
- 同域延迟显示改为测 `/index.html` 的 `HEAD` 请求

最后这一点也避免了依赖未暴露到公网的 `/healthz`。

## 8. 为什么保留“先落盘后校验”

### 问题

如果先校验再记录，就会丢掉失败请求的原始信息，后续排查很麻烦。

### 解决方案

保持当前策略不变：

- 所有请求先落盘
- 再校验
- 再决定摘要通知是否发送
- 原始日志通道保留完整上下文

## 9. 结论

这套方案最后稳定下来的关键不是某一个函数，而是三层一起配合：

- nginx 根路径分流
- 后端归一化与双通道通知
- 静态首页与接口同域共存

如果以后要复刻这套方案，最值得优先保留的也是这三块。
